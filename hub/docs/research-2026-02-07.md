# Research Notes (2026-02-07)

## Scope

Investigate patterns from OpenClaw/Agent-style ecosystems plus official Claude Code and Codex docs to design a robust outside-in developer hub.

## Key findings

### 1) Control plane + workers is a proven pattern

OpenClaw documents a single long-lived gateway (daemon) with multiple clients and nodes over a typed WebSocket protocol, plus explicit device trust/pairing and remote access via tunnels.

Design implication:
- Keep a central orchestrator (control plane) and separate execution workers.
- Treat mobile/remote interaction as client connections to the control plane.

Source:
- https://docs.openclaw.ai/concepts/architecture

### 2) Skill layering and precedence should be explicit

OpenClaw skills are loaded from multiple locations with a clear precedence order and load-time gating.

Design implication:
- Implement policy layering as `global -> repo -> module` with deterministic merge rules.
- Apply load-time trust checks for MCPs.

Source:
- https://docs.openclaw.ai/tools/skills

### 3) Compaction must be persistent and portable

OpenClaw describes compaction as summarizing older history into persistent session entries, keeping recent messages and a compact summary.

Design implication:
- Avoid relying on provider-specific conversation compaction.
- Write canonical checkpoints to disk (`checkpoint.latest.json`) for model-agnostic handoffs.

Sources:
- https://docs.openclaw.ai/concepts/compaction
- https://docs.openclaw.ai/concepts/memory

### 4) Claude Agent Teams are useful but expensive

Claude docs position Agent Teams as experimental, high-token parallel collaboration for tasks that benefit from teammate coordination; subagents are cheaper and better for focused delegation.

Design implication:
- Use teams only for parallel hypothesis/research/cross-layer work.
- Use subagents or single-session flow for sequential or tightly coupled edits.

Source:
- https://code.claude.com/docs/en/agent-teams

### 5) Codex officially supports MCP + worktrees + automation safety model

Codex docs define MCP server support in shared `config.toml`, plus app worktrees and automation guidance (sandbox/rules, background worktrees in Git repos).

Design implication:
- Keep MCP registry external to repositories by default.
- Execute unattended tasks in isolated worktrees with policy gates.

Sources:
- https://developers.openai.com/codex/mcp
- https://developers.openai.com/codex/app/worktrees
- https://developers.openai.com/codex/app/automations

### 6) Model routing should be capability-aware

Codex model docs recommend starting with `gpt-5.3-codex` for most coding tasks.

Design implication:
- Default to strongest coding model for complex tasks; route low-risk/repetitive jobs to lower-cost models.

Source:
- https://developers.openai.com/codex/models

## Architecture decisions for this repository

1. Build a chat-first CLI orchestrator.
2. Keep repository integration outside-in (no required installs in target repos).
3. Support per-repo policy and per-repo MCP registry as optional overlays.
4. Persist run artifacts and compact checkpoints independent of vendor session storage.
5. Gate every workflow with required checks and explicit risk reporting.
